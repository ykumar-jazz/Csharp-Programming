🌍 AWS Global Infrastructure

AWS Global Infrastructure is the physical backbone of AWS spread across the world.

It consists of:

Regions

Availability Zones (AZs)

Data Centers

Edge Locations / Points of Presence (PoPs)
-----------------------

1️⃣ AWS Regions
✅ What is a Region?

A Region is a geographical area that contains multiple data centers.

Examples of Region names:

us-east-1 (N. Virginia)

eu-west-3 (Paris)

ap-south-1 (Mumbai)

✅ Key Points

Each region is isolated from other regions

Most AWS services are region-scoped

You must select a region when deploying resources

Data does NOT automatically replicate across regions

✅ Why Regions Matter?

Compliance requirements

Latency (closer region = faster)

Disaster recovery strategy

Pricing differences

---------------------------
2️⃣ AWS Availability Zones (AZs)
✅ What is an Availability Zone?

An Availability Zone (AZ) is:

One or more data centers inside a Region.

Each Region has:

Minimum 3 AZs

Sometimes 6 or more

Example:

ap-south-1a

ap-south-1b

ap-south-1c

✅ Important Characteristics

AZs are physically separated

Connected with high-speed low-latency networking

Designed for fault isolation

✅ Why AZs Matter?

If one AZ fails:

Your application can still run in another AZ

Used for High Availability (HA)
------------------------------
3️⃣ AWS Data Centers
✅ What is a Data Center?

A Data Center is a physical building that contains:

Servers

Storage systems

Networking equipment

Power supply & cooling systems

Each Availability Zone has:

One or more Data Centers

✅ Important Point

AWS does NOT disclose exact locations for security reasons.

----------------------------
4️⃣ AWS Edge Locations / Points of Presence (PoPs)
✅ What are Edge Locations?

Edge Locations are:

Small data centers placed closer to users.

They are part of:

Amazon CloudFront (CDN)

Route 53

AWS Global Accelerator

✅ Purpose

Reduce latency

Deliver content faster

Improve user experience globally

Example:
If your server is in Mumbai but user is in Delhi:

Edge location in Delhi serves cached content

Faster than going to Mumbai every time


| Component         | What It Is                        | Purpose                      |
| ----------------- | --------------------------------- | ---------------------------- |
| Region            | Geographic area                   | Data residency & isolation   |
| Availability Zone | Multiple data centers in a region | High availability            |
| Data Center       | Physical building                 | Actual infrastructure        |
| Edge Location     | Global mini location              | Low latency content delivery |

IAM Policies Structure JSON

🔐 IAM Policy Structure – Step by Step

An IAM policy contains:

Version (Required)

Id (Optional)

Statement (Required – one or more)

✅ Step 1: Version (Always Required)

This defines the policy language version.

{
  "Version": "2012-10-17"
}

✔ Always use "2012-10-17"
✔ This is NOT a date of creation — it's policy language version.

✅ Step 2: ID (Optional)

Identifier for the entire policy.

{
  "Version": "2012-10-17",
  "Id": "S3-Account-Permissions"
}

✔ Optional
✔ Useful for tracking policies

✅ Step 3: Statement (Required)

Policy must contain at least one statement.

{
  "Version": "2012-10-17",
  "Statement": [
    {
      
    }
  ]
}

Statement is always inside an array [].

🔎 Step 4: Inside Statement

Each statement can contain:

Sid (Optional)

Effect (Required)

Principal (Required in resource-based policy)

Action (Required)

Resource (Required)

Condition (Optional)

🧩 Step 5: Full Example Explained
{
  "Version": "2012-10-17",
  "Id": "S3-Account-Permissions",
  "Statement": [
    {
      "Sid": "1",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:root"
      },
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::mybucket/*"
    }
  ]
}
🔍 Field-by-Field Explanation
🔹 Sid (Statement ID)
"Sid": "1"

✔ Optional
✔ Identifies individual statement

🔹 Effect
"Effect": "Allow"

Two possible values:

"Allow"

"Deny"

🚨 Explicit Deny always overrides Allow.

🔹 Principal (Used in Resource-Based Policy)
"Principal": {
  "AWS": "arn:aws:iam::123456789012:root"
}

✔ Defines WHO can access
✔ Required in S3 bucket policies, trust policies
✔ Not required in identity-based policies

🔹 Action
"Action": [
  "s3:GetObject",
  "s3:PutObject"
]

✔ Defines WHAT action is allowed
✔ Can be single or array
✔ Wildcard allowed:

"s3:*"
🔹 Resource
"Resource": "arn:aws:s3:::mybucket/*"

✔ Defines WHICH resource
✔ Uses ARN format
✔ "*" means all resources

🔹 Condition (Optional but Powerful)

Example:

"Condition": {
  "IpAddress": {
    "aws:SourceIp": "203.0.113.0/24"
  }
}

✔ Restricts policy based on condition
✔ Can restrict by:

IP address

MFA

Time

Region

Tags

🎯 Important Exam Notes
Field	Required?
Version	✅ Yes
Statement	✅ Yes
Effect	✅ Yes
Action	✅ Yes
Resource	✅ Yes
Principal	Only for resource-based
Condition	Optional
Sid	Optional
Id	Optional
🧠 Simple Memory Formula

IAM Policy =

Version
→ Statement
→ Effect
→ Action
→ Resource
→ (Principal if resource-based)
→ (Condition optional)

EBS Snapshots 

📦 EBS Snapshots
What is an EBS Snapshot?

An EBS Snapshot is a backup of your EBS volume at a specific point in time.

It allows you to restore your data or create new volumes from the backup.

Key Points

You can create a backup (snapshot) of your EBS volume at any time.

It is not required to detach the volume from the EC2 instance to create a snapshot.

However, detaching the volume is recommended for data consistency.

Snapshots can be copied across Availability Zones (AZs) or Regions.

How Snapshot Works

Example from the diagram:

An EBS volume (50 GB) exists in
US-EAST-1A Availability Zone.

A snapshot is created from this volume.

The snapshot is stored in Amazon S3 internally (managed by AWS).

You can restore the snapshot to create a new EBS volume.

The new EBS volume can be created in another AZ like
US-EAST-1B.

Simple Flow

EBS Volume → Snapshot → Restore → New EBS Volume

Why Use Snapshots?

Backup important data

Disaster recovery

Migrate volumes across AZ or Regions

Create identical volumes quickly

Example

Original volume:

EBS Volume: 50 GB
Region: us-east-1a

After snapshot restore:

New EBS Volume: 50 GB
Region: us-east-1b

Auto Scalling Group -- configuration and strategies

1. Manual Scaling: update the size of ASG manually 
2. Dynamic Scaling:
      Simple/Step Scalling 
	    When cloudwatch alarm is trggered(cpu>70%) then add one more instance
		When cloudwatch alarm is trggered(cpu<30%) then remove one instance
3. Target Tracking Scalling 
    EG:- I want the avg ASG CPU to stay around 40%
4. Scheduled Scaling 
    Anticipate a scaling based on known usage patterns 
    EG:- increase the min. capacity to 10 at 5 pm on friday
5. Predictive Scaling

Uses Machine Learning to predict future traffic ahead of time.

Automatically provisions the right number of EC2 instances in advance.

When to Use Predictive Scaling

Useful when your workload has predictable time-based patterns.

Example patterns:

Daily traffic spikes

Weekend traffic increases

Business hours traffic

How It Works

AWS analyzes historical metrics (like CPU usage or request count).

It predicts future demand.

It launches EC2 instances before the traffic spike happens.

Example Scenario
Morning traffic low → 2 instances
Lunch traffic spike → 6 instances
Night traffic low → 2 instances

Predictive scaling will prepare instances before the spike.

Key Benefit

Instead of reacting after traffic increases, AWS scales your infrastructure in advance.

This improves:

Performance

What’s serverless?

Serverless is a new paradigm in which the developers don’t have to manage servers anymore…

They just deploy code

They just deploy… functions!

Initially… Serverless == FaaS (Function as a Service)

Serverless was pioneered by AWS Lambda but now also includes anything that’s managed: “databases, messaging, storage, etc.”

Serverless does not mean there are no servers… it means you just don’t manage / provision / see them
